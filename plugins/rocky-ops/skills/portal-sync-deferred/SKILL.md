# 外部ポータル巡回更新(保留)

## これは何か
「食べログとぐるなびとHPを一括で店休にしといて」「臨時休業をポータル全部に反映して」と言われたときに、rocky-regiが正本(営業時間/コース/写真/臨時休業)を持っているのだから外部ポータルの管理画面まで代行入力して二重入力をなくしたい——という店長の要望に応えるためのスキル。ただし現状は着火しない。食べログ/ぐるなび等の各社利用規約はBOTによる自動操作を禁止しており、DOM操作型の代行入力は規約リスクが実装価値より重い。よってこのスキルは「保留中」と正直に返し、有効化後の想定手順だけを設計として残す。営業訴求で「レジが自動でポータルを更新する」と言わないための歯止めでもある。

## 使うMCPツール(rocky-regi)
- get_store_profile — rocky側の店舗正本(営業時間・住所・電話・臨時休業設定)。差分検出の起点。
- get_menu_and_courses — 正本のメニュー/コース。ポータル側コース情報との差分検出に使う。
- list_visibility_schedules — 期間限定の公開/非公開スケジュール。臨時休業・時短の元ネタ。
- get_active_visibility_now — 今この瞬間の可視状態。速報系の反映判断に使う。
- list_stores — 対象店舗の特定と、ポータル契約有無のメタ確認。
- Claude in Chrome(外部・rocky-ops連結) — 各ポータル管理画面のDOM操作担当。現状は保留のため呼ばない。
- Meta公式Ads MCP / Uber Eats等の正規API — 正規統合がある先はこちらに寄せる想定。**未接続**。契約後に .mcp.json 追加。

## 手順(この順で判断する)
### 1. まず「保留中」を伝える
呼ばれた瞬間に、このスキルが保留中であることを店長/オーナーに明示する。実行しない。金額や作業量の見積もりに進む前に、「今は自動反映できない。法務レビュー後に再開する」とだけ返す。ここで止めるのが最重要のガード。

### 2. なぜ止まっているかを一行で説明する
「食べログ/ぐるなび等はBOTによる自動操作を利用規約で禁止しているため、DOM操作での代行入力は規約リスクが高い」——このレベルの平易な説明にとどめる。法解釈はしない・約款文言の引用もしない。

### 3. 代替として「差分レポート」だけは提示できることを伝える
DOM操作せず、rocky正本と店長がヒアリングで教えてくれた各ポータルの現状値を突き合わせて、「食べログは営業時間が旧のまま/ぐるなびはコースAが載っていない/HPは臨時休業未反映」といったチェックリストを人間が手で更新するためのメモとして出すのは今も可能。ただしこれは本スキルの主目的ではないので、店長が望む場合のみ提示する。

### 4. 有効化後の想定手順(設計としてだけ残す・今は実行しない)
1) rocky正本を取得(get_store_profile / get_menu_and_courses / list_visibility_schedules / get_active_visibility_now)。
2) 各ポータルの現状値を取得。正規API先はAPI経由・それ以外は人手入力での取り込み。
3) 差分を検出(営業時間/臨時休業/コース/写真)。
4) 正規API連携先(Uber Eats等)は自動更新候補・BOT禁止先(食べログ/ぐるなび等)は差分レポート出力のみ、と経路を分ける。
5) いずれの経路でも、反映前に人間publish承認を必ず挟む。オーナー確認なしに外部反映しない。

### 5. スコープの外に出たら別スキルに渡す
「rocky側の臨時休業だけ登録したい」「モバイルオーダーの公開だけ止めたい」はこのスキルの対象外。rocky内で完結する話は該当スキル(可視性スケジュール系)に渡す。ポータル側だけの話も対象外(BOT禁止先はそもそも触らない)。

### 6. 営業訴求への転用を止める
このスキルの存在を根拠に「レジが食べログを自動更新します」「AIがポータル巡回します」といった外向け表現を作らない。法務レビュー前は禁止。

## 出力の型(表現ゆれを防ぐため固定)
呼ばれたときは、次の形で返す。金額や試算は返さない。

```
このスキル(外部ポータル巡回更新)は現在【保留中】です。

理由: 食べログ/ぐるなび等の外部ポータルはBOTによる自動操作を利用規約で禁止しているため、DOM操作での代行入力は規約リスクが実装価値より重い、と判断されています。

今できること:
- rocky側の正本(営業時間・臨時休業・コース)は通常どおり更新できます。
- ご希望があれば、rocky正本と各ポータル現状値の【差分チェックリスト】を出力できます(反映はお店側で手動)。

再開の条件:
- 法務レビューを通過し、正規API連携先(例: Uber Eats)とBOT禁止先(例: 食べログ/ぐるなび)を分けた再設計が承認された時点で再開します。
```

差分チェックリストを出す場合のみ、店舗名・項目(営業時間/臨時休業/コース/写真)・rocky正本の値・ポータル側の現状値・推奨アクション(人手更新)を素直な表で並べる。SSRの参考シナリオ止まりで、断定・予言はしない。UIウィジェットは使わない(誇張になるため)。

## やってはいけないこと(ガード)
- **DOM操作を実行しない**。Claude in Chromeでの食べログ/ぐるなび等の自動入力は保留中は着火しない。
- 「レジが自動でポータル更新します」「AIがポータル巡回します」といった営業訴求に転用しない。法務レビュー前は禁止。
- 各社利用規約の文言を引用したり法解釈をしたりしない。「BOT禁止条項に抵触しうる」という水準の説明にとどめる。
- 正規API連携先(Uber Eats等)についても、**未接続**の状態で「連携済み」と口走らない。契約・.mcp.json追加が済むまでは想定手順として残すだけ。
- rocky側の書込系(update_inventory / create_store_promo / create_visibility_schedule 等)に勝手に横滑りしない。ポータル一括更新の代替として無関係な書込を提案しない。
- 反映の前に必ず人間publish承認を挟む前提を崩さない。オーナー確認なしに外部反映につながる操作は提案しない。
- レート制限や連続叩きの話に踏み込まない。そもそも実行しないので発生しない。実行フェーズに入る場合は再設計時に別途定義する。
- 「保留中なのに一部だけこっそり動かす」抜け道を作らない。呼ばれたら保留と返す・差分レポートまで、が上限。
