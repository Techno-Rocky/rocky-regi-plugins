# レジ締め・日次クローズ

## これは何か
店長が「今日の営業終わり、締めたい」と言ったとき、いきなり close_business_day を叩いて締めない。締めは締めた行が immutable になる不可逆操作なので、まず「まだ会計していない卓」「まだ締めていないドロワー」が残っていないかを確認してから順に進める。現金の理論在高と実査額の差異は事実として提示し、丸めたり握りつぶしたりしない。オープン・ドロワー開閉・クローズ・締め帳票を、状況を読んで正しい順序で多段に回すための日次クローズ中核フロー。単発の締めコマンドでなく、残務点検→差異確認→確認ゲート→締めまでを閉ループで回す。

## 使うMCPツール(rocky-regi)
- get_current_business_day — 読取。今レジが開いている営業日(open/closed)。全ての起点。
- list_business_days — 読取。過去の営業日一覧。締め漏れ(open のまま残った日)の検出に使う。
- open_business_day — 実行系(書込)。営業日を開ける。閉じている店を開局する起点。要確認。
- close_business_day — 実行系(書込・不可逆)。営業日を締める。締めた行は immutable。最後の締め。要確認。
- list_cash_drawers — 読取。ドロワーの一覧と各開閉状態。締め対象の把握。
- get_open_cash_drawer — 読取。今開いているドロワーと理論在高。現金確認の主軸。
- open_cash_drawer — 実行系(書込)。ドロワーを開ける(釣銭準備金の登録)。要確認。
- close_cash_drawer — 実行系(書込・不可逆)。ドロワーを締める(実査額の確定)。締めた行は immutable。要確認。
- get_business_summary — 読取。当営業日の売上/客数/客単価。締め前の全体像の突き合わせ。
- list_payments — 読取。支払明細(現金/カード/QR等)。理論現金在高の裏取り。
- show_payments — 読取。支払のリッチUI表示。差異の根拠を見せるとき。
- print_closing_report — 実行系(印刷)。締め帳票(精算レポート)を出す。締め確定後の記録出力。

## 手順(この順で多段に呼ぶ)

### 1. 何をしたいのか意図を確定する
「開けたい(開局)」なのか「締めたい(クローズ)」なのかを最初に切り分ける。「営業日開けて」なら手順2→開局系(手順9)へ、「レジ締めたい」「今日終わり」なら締め系(手順2〜8)へ進む。「ドロワー確認」だけなら読取のみ(手順3)で止め、締めまで走らせない。曖昧なときは締め(不可逆)を先走らせず、まず現状を読んで確認する。

### 2. 今の営業日と締め漏れを確認する
get_current_business_day で今開いている営業日を取る。あわせて list_business_days で過去に open のまま締め忘れた日が無いか点検する。締め漏れが見つかったら「◯月◯日が開いたまま残っています」と提示し、当日締めと切り分けて確認を取る(古い日を勝手に締めない)。営業日が既に closed なら「今日は既に締め済みです」と返し、二重締めに進まない。

### 3. ドロワーの状態と理論在高を読む(読取のみ)
list_cash_drawers で全ドロワーの開閉状態を把握し、get_open_cash_drawer で今開いているドロワーの理論現金在高(釣銭準備金＋現金売上−現金払出)を取る。ここまでは全て読取で、まだ何も締めない。複数ドロワーがあれば1台ずつ状態を控える。

### 4. 未会計・未締めの残りを洗い出す(締め前ゲート)
get_business_summary で当営業日の売上・客数の全体像を取り、list_payments で支払明細を突き合わせる。会計が途中の卓や未処理の支払が残っていないかを確認する。残務があれば「まだ会計していない卓/未確定の支払が残っています」と具体的に提示し、締めに進まず先にそれを片付けてもらう。ここを飛ばして締めない(締めは不可逆)。

### 5. 現金差異を事実として算出する(丸めない)
get_open_cash_drawer の理論在高と、店員が数えた実査額(人間から金額を受け取る)を突き合わせる。差異(過不足)を「理論 ¥X / 実査 ¥Y / 差異 ¥Z(過剰 or 不足)」の形でそのまま提示する。差異をゼロに丸めない・良い方に寄せない・原因を勝手に断定しない。差異が出たら list_payments / show_payments で現金支払の内訳を見せ、事実として原因の当たり所(釣銭誤り/記録漏れ等の候補)だけ添える。

### 6. ドロワー締めの確認を取る(1操作ずつ・不可逆)
close_cash_drawer は締めた行が immutable になる不可逆操作。実行前に「対象ドロワー / 理論在高 / 実査額 / 差異 / 締めると後から金額修正できないこと」を提示し、人間の明示確認を取る。承認が出た1台だけ close_cash_drawer を実行する。複数ドロワーを承認1回でまとめて締めない。1台締めるごとに次を提示して確認を取り直す。

### 7. 全ドロワー締め後に営業日を締める(最後・不可逆)
全ドロワーが締め終わったことを list_cash_drawers で確認してから、close_business_day に進む。営業日締めも不可逆(締めた行は immutable)。実行前に「当営業日の売上サマリ / 締め済みドロワー一覧 / 締めると当日データが確定し後から変更できないこと」を提示し、人間の明示確認を取る。承認後に close_business_day を1回だけ実行する。ドロワーが1台でも開いたまま残っていたら締めず、手順6へ戻る。

### 8. 締め帳票を出す(締め確定後)
close_business_day が通ったら print_closing_report で締め帳票(精算レポート)を出力する。帳票は締め確定後の記録なので、締め前に先出ししない。プリンタ不達(用紙切れ等)が出たら「帳票印刷に失敗しました。締め自体は完了しています」と切り分けて返し、締めをやり直させない(締めは既に不可逆に確定している)。

### 9. 開局(「営業日開けて」のとき)
締めでなく開局を頼まれたら、get_current_business_day で二重開局にならないか確認する。既に open なら「今日は既に開いています」と返して何もしない。closed で今日まだ開いていなければ、open_business_day の前に釣銭準備金(開始現金)の額を人間に確認し、提示して承認を取ってから open_business_day を1回実行する。続けてドロワーを開けるなら、準備金額を確認のうえ open_cash_drawer を1操作ずつ実行する。開局を勝手に連鎖実行しない。

## 出力の型(表現ゆれを防ぐため固定)

### ケースA: 締め前サマリ(確認ゲート)
```
【本日の締め前確認】◯月◯日(◯曜)
売上: ¥X,XXX,XXX / 客数: XX / 客単価: ¥X,XXX
未会計の卓: N件(あれば内訳 / なければ「なし」)
未確定の支払: N件(あれば内訳 / なければ「なし」)

ドロワー現金:
- ドロワー◯: 理論 ¥X / 実査 ¥Y / 差異 ¥Z(過剰 or 不足 or 差異なし)

締めると当日データとドロワー金額は確定し、後から修正できません。
締めてよいか確認してください(1台/1日ずつ実行します)。
```

### ケースB: 差異ありの提示(丸めない)
```
ドロワー◯ に現金差異があります。
理論在高 ¥X / 実査額 ¥Y / 差異 ¥Z(不足 or 過剰)
現金支払の内訳: <list_payments の現金分を1〜2行>
考えられる当たり所: <釣銭誤り / 記録漏れ 等の候補のみ。断定しない>
差異はそのまま記録されます。締める前に現場で照合してください。
```

### ケースC: 締め漏れ検出 / 二重操作の抑止
```
◯月◯日が open のまま残っています(本日ではありません)。
本日の締めとは別件です。この日を締めるか、本日分だけ締めるか確認してください。
```
（既に締め済みなら「本日は既に締め済みです」、開局要求で既に open なら「本日は既に開いています」と返す)

差異や支払内訳をリッチに見せるときだけ show_payments を添える。締め前サマリは数字で簡潔に。

## やってはいけないこと(ガード)
- **close_business_day / close_cash_drawer は不可逆(締めた行は immutable)**。実行前に必ず「対象・理論/実査・差異・戻せないこと」を提示し、人間の明示確認を取ってから1操作ずつ実行する。承認1回で複数ドロワー/複数日をまとめて締めない。
- **未会計・未締めの残りを確認せずに締めない**。get_business_summary / list_payments で残務を洗ってから締めに進む。締めた後は直せない。
- **現金差異を丸めない・握りつぶさない**。理論と実査の差はそのまま事実として提示する。ゼロに寄せない、良い方に寄せない、原因を勝手に断定しない。
- **ドロワーが1台でも開いたまま営業日を締めない**。全ドロワー締め完了を list_cash_drawers で確認してから close_business_day に進む。
- **二重締め・二重開局をしない**。get_current_business_day で状態を確認し、既に closed なら締めない、既に open なら開けない。締め漏れの古い日を当日締めと混ぜて勝手に締めない。
- **締め帳票を締め前に先出ししない**。print_closing_report は close_business_day 確定後の記録出力。印刷不達は締めと切り分け、締めをやり直させない(締めは既に確定済み)。
- **開局時の釣銭準備金・締め時の実査額は人間から受け取る**。金額を推測で埋めない。準備金/実査額の確認なしに open_cash_drawer / close_cash_drawer を走らせない。
- **権限(capability)で弾かれた締め操作(422)を無理に通さない**。締め権限を持つ担当者へエスカレする旨を返す。
- **UI/店員向け出力に物理名・enum値・内部ジャーゴンを出さない**。「ドロワー」「理論在高」「実査額」「差異」など役割別の自然な日本語で返す。
- **レジのAIが自動で締める等の誇張をしない**。締めの判断と現金実査は人間、レジはデータ供給と受け皿に徹する。予言・断定をしない。
- **レート制限を超える反復呼び出しをしない**。同じ営業日サマリ/ドロワー状態を何度も再問い合わせしない。読取は各1回で足りる。
