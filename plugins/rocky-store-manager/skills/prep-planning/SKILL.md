# 明日の仕込み量の見立て

## これは何か
店長が「明日どれくらい仕込む?」と聞いたとき、需要予測ツールの数字を1つ持ってきて「◯人来ます」と言い切らない。来客のベースライン予測に、同条件(曜日/天候/季節/祝日)の類似営業日の実績レンジ、過去の季節性、品目ごとの売れ方、そして今の在庫を重ねて、品目別の仕込み量を幅(下限〜上限)で出す。予測はあくまで統計的な当たりであって予言ではない。類似日が少ない・信頼度が低いときは「判定不能・幅で示す」と正直に返す。生ものや廃棄リスクの高い品は上限を抑えめに助言する。これは読み取り専用で、発注も在庫更新もしない。単一ツールの値を鵜呑みにする過ちを、多段の重ね合わせで防ぐためのスキル。

## 使うMCPツール(rocky-regi)
すべて読み取り専用。このスキルは書き込み・実行系を一切呼ばない。
- get_demand_forecast_baseline —（読取専用）来客/売上のベースライン予測と信頼度。入口の主軸。単独では鵜呑みにしない。
- show_demand_forecast —（読取専用）ベースライン予測をリッチUIで提示。数字の裏取りを人が見る用。
- get_similar_business_days —（読取専用）同条件(曜日/天候/季節/祝日)の類似営業日から実績レンジ(mean±std)と信頼度(confidence)。予測を実績で挟む主軸。
- get_demand_history_summary —（読取専用）過去履歴の季節性・トレンド。ベースラインが季節の山谷を織り込めているかの検算。
- get_menu_engineering —（読取専用）品目の売れ方(4象限・出数)。来客を品目別に割るための出数比の素材。
- get_hourly_heatmap —（読取専用）曜日×時間帯の山。ピーク帯と仕込みの投入タイミングの目安。
- list_inventory —（読取専用）現在庫。仕込み必要量から差し引く在庫の確認。
- list_stores —（読取専用）店舗が明示されないときの対象店の確定(手順1)。

## 手順(この順で多段に呼ぶ)
順序に意味がある。ベースライン単独で答えを出さず、実績レンジで挟み、季節性で検算し、品目に割り、在庫で差し引く。この重ね合わせがこのスキルの核。

### 1. 対象日と店舗を確定する
「明日」「今週末」「◯月◯日」を1営業日に確定する。営業日は暦日でなくレジの当営業日基準。店舗が明示されない場合は list_stores とユーザー所属から1店に絞る。複数店舗を同時に混ぜない(単店の仕込み見立て)。対象日の属性(曜日・祝日か・季節・分かれば天候見込み)をここで押さえる。以降の類似日マッチングの前提になる。

### 2. ベースライン予測を取る(入口・ただし単独では信じない)
get_demand_forecast_baseline で対象日の来客/売上のベースラインと信頼度を取る。これは入口の当たりであって、まだ答えではない。ここで得た数字を単独で店長に投げない。次のステップで実績レンジと突き合わせて初めて使える値になる。

### 3. 同条件の類似営業日で挟む(核心・ここで幅が決まる)
get_similar_business_days で同曜日・天候区分・季節・祝日属性の類似営業日群から、来客/売上の mean±std と confidence を得る。ベースライン予測(手順2)がこの実績レンジの内側に収まっているかを確認する。

- ベースラインが実績レンジ内 → ベースラインを中心、レンジ幅(mean-1σ〜mean+1σ)を仕込みの下限〜上限の土台にする。
- ベースラインが実績レンジを外れている → ベースラインを鵜呑みにせず、実績レンジ側を優先して幅を広めに取る。乖離の事実を出力に明記する。
- **confidence が低い/類似日が少ない(INSUFFICIENT_SAMPLE 相当) → σそのものが信用できない。手順7の判定不能テンプレで「幅で示す・断定しない」と正直に返す。深掘りの品目割りには進むが、数量は必ず幅かつ低めに寄せる。**

### 4. 季節性で検算する(ベースラインの山谷ズレを拾う)
get_demand_history_summary で過去履歴の季節性・トレンドを見て、対象日が繁忙期の入り/端境期/連休前後などの山谷にあたるかを確認する。ベースライン(手順2)と実績レンジ(手順3)がこの季節の山谷を織り込めているかを検算する。織り込めていない気配があれば、幅の上限/下限をその方向に寄せる。ここは数値を上書きするのでなく、幅の寄せに使う。

### 5. 品目別の出数比に割る
get_menu_engineering で品目ごとの出数と4象限を取り、直近の「来客あたりその品が何食出るか」の比率(出数比)を素材にする。手順3で確定した来客の幅に出数比を掛けて、品目別の必要食数を幅で出す。スター品(高人気×高粗利)は欠品させたくないので下限を切り上げ、負け犬・死に筋は仕込みを絞る側に寄せる。出数比は直近実績からの当たりであって固定係数ではない旨を押さえる。

### 6. ピーク帯と在庫で最終補正する
get_hourly_heatmap で対象日相当のピーク帯(ランチ/ディナー)を確認し、仕込みをどの時間帯までに何割用意すべきかの目安を添える(数量そのものより投入タイミングの助言)。続いて list_inventory で現在庫を確認し、手順5の品目別必要食数から在庫を差し引いて「追加で仕込む量」を幅で出す。ここで在庫が必要量を上回る品は「仕込み不要・在庫消化優先」と返す。**生もの・廃棄リスクの高い品は、下限寄りの数量を推奨し上限は抑えめに助言する**(過剰仕込みは廃棄に直結するため)。

### 7. 判定不能テンプレ(手順3で confidence 低・類似日僅少のとき)
リッチUIで気の利いた確定値を出さず、テキスト中心で「幅で示す・断定しない」を明示して返す。ベースラインは参考値、品目別は低め寄せの幅のみ。「同条件の実績が少ないので、この数量は確度が低い当たりです」と正直に添える。断定・予言をしない。

## 出力の型(表現ゆれを防ぐため固定)

### ケースA: 通常(confidence 十分)
```
◯月◯日(◯曜・◯シーズン)の来客見込み: 約 X〜Y 人
  ベースライン予測: 約 Z 人(信頼度 ◯)
  同条件の類似営業日: ◯件・confidence=◯ / 実績レンジ X〜Y 人(平均 A ± B)
  季節性: <繁忙入り/端境/連休前後 等・幅をどちらに寄せたか>

品目別の仕込み量(来客 × 出数比 − 現在庫、幅で表示):
  ◯◯(スター): 追加 P〜Q 食  ※欠品回避で下限切り上げ
  △△: 追加 R〜S 食
  □□(生もの): 追加 T〜U 食  ※廃棄リスクのため上限抑えめ推奨
  ××: 仕込み不要(現在庫で足りる・消化優先)

投入タイミング(ピーク帯): <◯時までに◯割、等>
```

### ケースB: 判定不能(confidence 低・類似日僅少)
```
同条件(◯曜・天候◯・◯シーズン)の類似営業日が◯件しかなく、来客見込みを断定できません(confidence=低)。
ベースライン予測 約 Z 人は参考値です。

品目別は確度の低い当たりとして、幅かつ低め寄せで示します:
  ◯◯: 追加 P〜Q 食(低め寄せ)
  □□(生もの): 追加 T〜U 食(廃棄リスクのため最小限)

サンプルが揃うまで、当日の立ち上がりを見て追い仕込みで調整するのが安全です。
```

数量はすべて幅(下限〜上限)で示す。単一の断定値を出さない。リッチUIで補足するときは show_demand_forecast(ベースライン)と show_hourly_heatmap(ピーク帯)を添える。数字だけの長文にしない。

## やってはいけないこと(ガード)
- **需要予測を予言・断定にしない**。get_demand_forecast_baseline の値を「明日は◯人来ます」と言い切らない。統計的な当たりであり、必ず幅で示す。
- **ベースライン(手順2)単独で答えを出さない**。get_similar_business_days の実績レンジで挟み、get_demand_history_summary で季節性を検算してから初めて数量を出す。順序を飛ばさない。
- **confidence が低い/類似日僅少を、十分なときと同じ強さで断定しない**。「判定不能・幅で示す・低め寄せ」と正直に返す。INSUFFICIENT_SAMPLE を握りつぶして確定値をでっち上げない。
- **生もの・廃棄リスクの高い品の上限を強気に出さない**。過剰仕込みは廃棄に直結する。下限寄りを推奨し、当日の追い仕込みでの調整を促す。
- **出数比を固定係数のように扱わない**。直近実績からの当たりであり、来客の幅に掛けた結果もあくまで幅。単一値に丸めない。
- **このスキルで在庫を触らない・発注しない**。読み取り専用。update_inventory や import_csv などの書き込みはここから叩かない。在庫是正・発注は別スキル + オーナー確認に渡す。仕込みを「仕入れの自動発注」と言い換えない。
- **在庫が足りている品を仕込ませない**。list_inventory で現在庫が必要量を上回る品は「仕込み不要・消化優先」と返す。惰性で全品を仕込み対象にしない。
- **レート制限を超える反復呼び出しをしない**。横断7ツールは各1回まで。同一日を何度も再問い合わせしない。品目ごとに個別ツールを回すスイープはしない。
- **UI文言に物理名・enum・内部ジャーゴンを出さない**。confidence 区分や象限名はそのまま出さず、「信頼度」「よく出る品/絞る品」など店長の言葉に直す。
- **「レジAIが自動で仕込み量を決める」等と誇張しない**。数量はあくまで見立ての当たりで、最終判断は店長。予言・自動化を装わない。
- **外部の天候API・仕入れ先システム連携は未接続**。天候は類似日マッチングと分かる範囲の見込みに留め、外部連携があるかのように書かない。
