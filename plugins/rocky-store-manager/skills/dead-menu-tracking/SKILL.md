# 死に筋・失速メニュー追跡(方向と併売で切り分ける)

## これは何か
店長が「最近売れてない商品ある?」「どこテコ入れすればいい?」と聞いたときに、動かない商品リストを羅列して終わりにしない。まず「元から動いていない(構造的死に筋)」のか「前は動いていたのに落ちた(失速)」のかを、前期間比の販売数で明示的に切り分ける。次に併売相手の有無で"単品で撤去できるか/抱き合わせでまだ動かせるか"の文脈を添え、値下げで戻せるかを試算し、最後に打ち手[差し替え/プロモ/撤去]まで導く。売上レビューから降りてくる連結先であり、季節プロモ差し替えスキルへの入口。

## 使うMCPツール(rocky-regi)
- get_store_insights — target=loss/engineering の自動気づきを取り、追う品目の当たりをつける。severity=warning を優先。
- list_dead_menu_items — 一定期間動いていない品目のリスト。構造的死に筋の一次候補。
- get_menu_engineering — 4象限(Star/Plow Horse/Puzzle/Dog)。Dog と Puzzle を対象に落とす。
- get_menu_abc_analysis — C 帯(下位寄与)を、死に筋候補と重ねて確認。
- get_basket_analysis — 対象品の併売相手の有無・強さ。「単品死に筋」か「セットで動いている」かを判別。
- list_cancelled_items — キャンセルが多い=注文はされるが提供できていない失速の別解釈を潰す。
- simulate_price_change — 値下げ耐性の**参考シナリオ**(読取専用・価格は変えない・需要予測ではない)。
- show_menu_engineering / show_menu_abc / show_basket_analysis / show_cancelled_items — リッチUIで描画。

## 手順(この順で多段に呼ぶ)

### 1. 対象期間と比較期間を決める
今期=直近4週、比較期=その前の4週を既定にする。店長が「先月」「今月」と言えばそれに合わせる。期間が短すぎる(2週未満)ときは「サンプル不足」と明示し、結論を弱める。

### 2. 気づきから当たりをつける
まず get_store_insights(target=loss) と get_store_insights(target=engineering) を呼ぶ。severity=warning の品目を優先候補にする。気づきゼロなら「自動検知では要注意なし」と正直に書き、次の網羅走査に進む(ゼロを"好調"と言い換えない)。

### 3. 一次リストを取る(構造的死に筋の候補)
list_dead_menu_items で当期の非稼働品を取る。ここは「今期に売れていない」だけで、失速かどうかはまだ分からない。

### 4. 失速か構造的死に筋かを明示的に判別する(このスキルの肝)
候補ごとに前期→今期の販売数の方向を出し、次の閾値で分類する(明文化して出力に載せる):

- **構造的死に筋**: 前期販売数 ≤ 3 かつ 今期販売数 ≤ 3(元から動いていない)。
- **失速**: 前期販売数 ≥ 10 かつ 今期販売数 が前期の 50% 未満(前は動いていたのに落ちた)。
- **緩速**: 上のどちらでもない下降(前期比マイナスだが失速閾値未満)。方向のみ添えて打ち手は保留。
- **サンプル不足**: 前期・今期のどちらかが 3 未満で判定できない。**判定不能と明示**し、断じない。

get_menu_engineering の象限(Dog/Puzzle)と get_menu_abc_analysis の C 帯を裏付けに紐づける。

### 5. キャンセル起因の"見せかけ失速"を潰す(遅延評価)
失速判定が出た品目だけ list_cancelled_items を見る。キャンセル比率が高ければ「売れていない」ではなく「注文はされたが提供できていない(品切れ・提供中止・オペミス)」の可能性を書き添え、打ち手を差し替え/プロモから**オペ改善**に振り替える。

### 6. 併売相手の有無で"抱き合わせで動かせるか"を見る
失速・構造的死に筋の両方について get_basket_analysis を呼び、対象品の併売相手を確認する:

- **併売相手あり(強)**: 単品では動かないがセットで出ている → 撤去は保留、抱き合わせ強化 or セット化の余地。
- **併売相手なし**: 単独で動いていない → 差し替え・撤去の対象として妥当。

### 7. 値下げ耐性を試算する(失速のみ・参考シナリオ)
失速品目に限り simulate_price_change を呼び、値下げでの想定戻りを**参考シナリオ**として添える。**これは需要予測ではない・価格は変えない・断定しない**。構造的死に筋には呼ばない(元から需要がないため値下げでは戻らない)。

### 8. 打ち手[差し替え/プロモ/撤去]で締める
品目ごとに次のいずれかで締める。実行はこのスキルからは行わない:

- **差し替え**: 構造的死に筋・併売なし → メニュー編集(別スキル)。
- **プロモ**: 失速・併売相手あり、値下げ耐性 参考シナリオがプラス → 季節プロモ差し替えスキルへ渡す。
- **撤去**: 構造的死に筋・併売なし・在庫も動かない → メニュー編集(別スキル)。
- **オペ改善**: キャンセル比率が高い → 店長にオペ確認を促す。

## 出力の型(表現ゆれを防ぐため固定)

冒頭に判別閾値を1行で明示してから、品目ごとに1行:

判別閾値: 構造的死に筋=前期≤3 かつ 今期≤3 / 失速=前期≥10 かつ 今期が前期の50%未満

[品目名] | [失速/構造的死に筋/緩速/サンプル不足] | 前期○個 → 今期○個(方向: 下向き/横ばい) | 併売相手[有(相手品名)/無] | 値下げ参考シナリオ[+○% / なし / 対象外] | 打ち手: [差し替え/プロモ/撤去/オペ改善/保留]

末尾に1行:
> プロモ実行は季節プロモ差し替えスキルへ、メニュー差し替え・撤去はメニュー編集スキルへ渡してください。

リッチUIで描画するときは show_menu_engineering / show_basket_analysis / show_cancelled_items を併用する。

## やってはいけないこと(ガード)

- 動かない商品リストだけ出して終わらない。必ず失速か構造的死に筋かを閾値で切り分けて出す。
- 判別閾値を出力に載せずに「これは失速です」と断じない。閾値は毎回明示する。
- サンプル不足(前期・今期のどちらかが3個未満)を無理やり分類しない。「判定不能」と正直に書く。
- 自動気づきゼロを「メニューは健全」と言い換えない。「自動検知では要注意なし」までに留める。
- simulate_price_change の結果を「値下げすれば○%戻る」と断定しない。**参考シナリオ**と明示し、需要予測ではないことを添える。構造的死に筋には呼ばない。
- キャンセル比率が高い品目を「売れていない」と断じない。オペ改善に振り替える。
- 実行系(create_store_promo / update_store_promo / reorder_menu_items / update_inventory 等)はこのスキルからは**呼ばない**。プロモは季節プロモ差し替えスキル、メニュー差し替え・撤去はメニュー編集スキルに必ず渡す。人間の publish 承認を経ること。
- 併売相手の有無を確認せずに「撤去」と結論しない。セットで動いている品を単独判断で切ると、相手品の売上も落ちる。
- MCP を1品ずつループで叩いてレート制限を超えない。候補を絞ってから呼ぶ。
- Meta 公式 Ads MCP は未接続のため、広告連動の判断はこのスキルからは行わない。外部ポータル巡回(Claude in Chrome)は BOT 規約リスクで保留状態のため呼ばない。
