# 単店急落 原因究明(誤警報を握りつぶす探偵)

## これは何か
店長や本部が「昨日なんで落ちたの?」と聞いたときに、いきなり深掘りして「客数が減った」「あの商品が売れなかった」と答えを作らない。まず同条件(曜日/天候/季節/祝日)の類似営業日と比べて、そもそも異常なのかを確かめる。期待レンジ内なら「異常ではない」と根拠付きで断言してそこで終わる。異常なときだけ、逸失→時間帯→商品→顧客感情の順に絞り込み、主因1つに寄せて打ち手を出す。誤警報を上流で握りつぶすゲートでもあり、奪還配信など下流スキルの起動条件を守る役目を持つ。

## 使うMCPツール(rocky-regi)
- get_similar_business_days — 同条件(曜日/天候/季節/祝日)の類似営業日から期待レンジ(mean±std)と信頼度(confidence=low<4|medium4-7|high≥8)を返す。主軸。判定はここで決まる。
- get_business_summary — 対象日の実測売上/客数/客単価。期待レンジと突き合わせる材料。
- list_cancelled_items — 逸失(キャンセル・欠品・打ち直し)の内訳。ピーク帯逸失の主因候補。
- get_hourly_heatmap — 時間帯別の売上/客数。ピーク崩れ(通常山が立たない時間の落ち込み)を可視化。
- get_menu_engineering — スター/金の卵/問題児/負け犬の4象限。スター販売数の異常減を拾う。
- get_survey_summary — organicなCSAT/NPS。bias_split(招待/organic)を意識して自然回答側のスコア低下を見る。
- show_hourly_heatmap / show_cancelled_items / show_menu_engineering / show_survey_summary — 深掘りに入ったときだけリッチUIで根拠を示す。誤警報時は開かない。
- list_stores — 店舗が明示されないときの対象店の確定(手順1)。

## 手順(この順で多段に呼ぶ)

### 1. 対象日と店舗を確定する
「昨日」「先週の火曜」「◯月◯日」を1営業日に確定する。営業日は暦日でなくレジの当営業日基準(open→close)。店舗が明示されない場合は list_stores とユーザー所属から1店に絞る。複数店舗を同時に扱わない(単店急落フロー)。

### 2. 反実仮想の期待レンジを先に取る(ここが核心)
必ず深掘りより先に get_similar_business_days を呼ぶ。同曜日・天候区分・季節・祝日属性でマッチした過去営業日群から、売上のmean±std(1σ)と confidence(low/medium/high) を得る。ここを飛ばして list_cancelled_items 等に進んではいけない。

### 3. 異常かどうかを判定して分岐する(遅延評価)
- 実測が mean-1σ 以上 → 「異常ではない」。深掘りしない。ステップ7の誤警報テンプレでテキストのみ返して終了。ウィジェットも開かない、下流スキル(奪還配信/SSR/CS共有)も呼ばない。
- 実測が mean-1σ 未満 mean-2σ 以上 → 「弱い異常」。深掘りには進むが、断定は避け「要因の候補」として提示。
- 実測が mean-2σ 未満 → 「有意に下回る」。深掘りへ進む。
- confidence=low(類似日<4件) → σそのものが信用できない。判定不能として正直に返し、深掘りに進まない。「同条件のサンプルが◯件しかないため、異常かどうか判定できない」と明示する。

### 4. 深掘りは4本を順番に(異常判定が出たときだけ)
順序が主因帰属の根拠になる。順番を崩さない。
1. list_cancelled_items — 逸失金額と件数、時間帯、理由(在庫切れ/オーダー取消/差し戻し)。逸失が実測不足額の大部分を説明できるならここで主因確定に近づく。
2. get_hourly_heatmap — 通常のピーク時間(ランチ/ディナー)が立っているか。ピーク崩れの有無で「機会損失型」か「全時間帯低調型」を切り分ける。
3. get_menu_engineering — スター商品(高粗利×高販売数)の販売数が同条件平均より落ちているか。スター単品の異常減は真因になりやすい。
4. get_survey_summary — organic(自然回答)側のCSAT/NPSの直近低下。招待バイアス側は判断材料に使わない。感情起因(接客・待ち時間・味)を疑う材料。

### 5. 主因を1つに寄せる(帰属ルール)
複数シグナルが同時発火することは普通に起きる。曖昧に「複合要因」で逃げない。以下の優先で主因を1つに寄せ、他は「併発」と明示する。
- 時系列で先に起きたもの > 後に起きたもの(接客低下→キャンセル増、の順なら接客側が主因)
- 実測不足額に対する説明力(金額規模)が大きいもの > 小さいもの
- confidence=high の裏付けを持つもの > low のもの
どうしても寄せられない場合のみ「主因未確定・要現場ヒアリング」と正直に返す。作文で埋めない。

### 6. 真因に紐づく打ち手を1つだけ添える
主因ごとに直結する打ち手を1つ出す。複数並べない。実行系は含めない(このスキルは読み取り専用)。
- 主因=逸失(欠品) → 「該当商品の在庫是正」(update_inventory は別スキル/要オーナー確認)
- 主因=ピーク崩れ → 「該当時間帯の増員検討」(現場判断)
- 主因=スター販売減 → 「メニュー導線でスターを前面化」(reorder_menu_items は別スキル/要オーナー確認)
- 主因=CSAT低下 → 「organic低下を店舗内で共有する notice 作成」(create_notice は別スキル/要オーナー確認・人間publish承認)

### 7. 誤警報テンプレ(ステップ3で「異常ではない」判定のとき)
ウィジェットを開かず、テキストのみ・短く・根拠付きで返す。SSR/奪還配信など下流を絶対に呼ばない。ここで会話を閉じる。

## 出力の型(表現ゆれを防ぐため固定)

### ケースA: 異常ではない(mean-1σ以上)
```
◯月◯日(◯曜)の売上 ¥X,XXX,XXX は、同条件の類似営業日(◯件・confidence=◯)の期待レンジ ¥YYY 〜 ¥ZZZ(平均 ¥AAA ± ¥BBB)の内側です。
天候・曜日で説明できる範囲で、異常ではありません。深掘りは不要です。
```

### ケースB: 判定不能(confidence=low)
```
同条件(◯曜・天候◯・◯シーズン)の類似営業日が◯件しかなく、異常かどうか判定できません(confidence=low)。
実測 ¥X,XXX,XXX に対して期待レンジは参考値です。判断を保留します。
```

### ケースC: 有意に下回る(深掘り実施)
```
◯月◯日の売上 ¥X,XXX,XXX は、期待レンジ ¥YYY 〜 ¥ZZZ を¥VVV下回っています(confidence=◯)。

主因: <逸失/ピーク崩れ/スター販売減/CSAT低下 のいずれか1つ>
根拠: <該当ツールの数字を1〜2行で。金額・件数・時間帯を具体>
併発: <あれば1〜2件、なければ「なし」>

次の一手: <主因に直結する1つだけ>
```

show_hourly_heatmap / show_cancelled_items / show_menu_engineering / show_survey_summary は、ケースCで主因を裏付けるものだけを開く。ケースA・Bでは開かない。

## やってはいけないこと(ガード)

- 反実仮想検証(get_similar_business_days)を飛ばして深掘りに進まない。順番厳守。これがこのスキル最大の存在意義。
- 異常でないのに「落ち込んだ原因は◯◯です」と作文しない。誤警報を握りつぶすのが仕事。
- confidence=low を medium/high と同じ強さで断定しない。「判定不能」と正直に返す。
- 複数シグナル同時発火時に「複合要因です」で逃げない。時系列と金額規模で主因1つに寄せる。寄せられないなら「主因未確定」と正直に。
- ケースA/Bで show_* ウィジェットを開かない。下流の SSR・奪還配信・CS共有 notice スキルを呼ばない。ここで会話を閉じる。
- 天候・曜日で説明できる範囲を「絶好調」「深刻」と情緒的に言い換えない。期待レンジ内は「異常ではない」まで。
- 実行系(update_inventory / create_notice / reorder_menu_items / create_coupon_distribution_scenario など)はこのスキルから直接叩かない。打ち手は提案までで、実行は別スキル+オーナー確認+人間publish承認に渡す。
- SSR(参考シナリオ生成)を主因確定の根拠に使わない。SSRは参考止まり・予言禁止。
- レート制限を超える反復呼び出しをしない。深掘り4本は各1回まで。同一日を何度も再問い合わせしない。
- 外部ポータル巡回(Claude in Chrome での食べログ等の外部レビュー取得)はBOT規約リスクで保留中。このスキルからは呼ばない・言及しない。
- 外部Ads MCP(Meta公式)は未接続。広告出稿側の要因はこのスキルでは判定材料にしない。
