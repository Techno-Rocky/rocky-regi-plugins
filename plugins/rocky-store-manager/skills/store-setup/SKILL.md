# 店舗の初期設定ナビ(機器・出し分け・招待)

## これは何か
新しく作られた店舗を、実際に営業できる状態まで持っていくための伴走スキル。プリンタを1台登録して終わり、スタッフを1人招いて終わり、にしない。まず「今なにが残っているか」を取得し、それを軸に前提順序(機器を登録して疎通が通ってから出し分けを割り当てる、招待したら権限が意図通りかを確かめる)を守って、抜け漏れなく1つずつ進める。設定は書き込みなので、分析→提示→確認→実行の順を崩さず、人間の明示確認を取ってから1操作ずつ反映する。設定するたびに残タスクを取り直し、「何が終わって何が残っているか」を常に見せるのが役目。

**このスキルの範囲は、作成済みの店舗の機器・出し分け・ユーザーのセットアップに限定する。店舗そのものの新規作成はここでは扱わない**(オーナー登録ウィザードや管理側で行う)。店舗がまだ無い状態でこのスキルを起動しない。

## 使うMCPツール(rocky-regi)
- get_setup_steps — 残っている初期設定タスク一覧。このスキルの軸。読取専用。毎回ここに戻って残りを照合する。
- get_required_equipment — この店舗の稼働に必要な機器の一覧。足りない機器を洗い出す。読取専用。
- list_printers — 登録済みプリンタの一覧・状態。何が既に登録されているかの確認。読取専用。
- register_printer — プリンタを登録する(実行系・要確認)。
- test_printer — 登録済みプリンタへ疎通テストを送る(実行系・要確認)。物理的に紙が出るので確認後に1台ずつ。
- set_kitchen_station_printers — キッチンの調理場(ステーション)に出力先プリンタを割り当てる(実行系・要確認)。
- set_kitchen_station_category_routes — メニュー分類をどの調理場に流すかの出し分けを設定する(実行系・要確認)。
- invite_user — 役割を指定してスタッフを招待する(実行系・要確認)。
- list_invitations — 招待の送付状況・受諾状況の確認。読取専用。
- get_role_permissions — 役割ごとの権限内容の確認。招待した役割が意図通りかを裏取りする。読取専用。

## 手順(この順で多段に呼ぶ)

### 1. 残タスクを取得して現在地を確定する(ここが軸)
まず get_setup_steps を呼び、この店舗で残っている初期設定タスクを取得する。いきなり register_printer や invite_user に飛ばない。ここで得た「残り一覧」がこのスキル全体の進行台帳になる。ユーザーが「プリンタだけ」「招待だけ」と限定してきても、まず全体の残りを一度は照合し、依存関係(出し分けは機器登録の後、など)を崩さないようにする。店舗が確定していない場合は list_stores とユーザー所属から1店に絞る。複数店舗を同時にセットアップしない。

### 2. 必要機器を洗い出して不足を確認する
get_required_equipment でこの店舗の稼働に必要な機器を取得し、list_printers の登録済み一覧と突き合わせる。足りない機器があれば「登録が必要」として提示する。必要数を満たしていれば機器登録はスキップして手順5(出し分け)へ進んでよい。ここは読取だけ。まだ何も書き込まない。

### 3. プリンタを登録する(確認後に1台ずつ)
不足しているプリンタについて、登録する対象(設置場所・用途・接続先の識別情報)を人間に提示し、明示的な確認を取る。確認が出た1台だけ register_printer で登録する。複数台を一度に束ねて登録しない。1台登録するごとに list_printers で登録結果を確認し、次の1台に進む。接続先の識別情報が不確かなまま登録しない(誤った宛先の機器を作らない)。

### 4. 疎通テストを通す(前提ゲート・確認後に1台ずつ)
登録できたプリンタに対し test_printer で疎通テストを送る。**これは物理的にテスト用紙が出力される操作なので、「この機器にテスト印刷を送る」と対象を明示して確認を取ってから、1台ずつ送る**。テストが通らないプリンタ(応答なし・エラー)は「疎通NG」として記録し、**出し分けの割り当て対象にしない**。ここで通ったプリンタだけが次の出し分けの候補になる。テストが通らないのに先へ進めない。用紙切れ・接続不備が疑われる場合は現場対応を促し、通るまで出し分けには進まない。

### 5. キッチンの出し分けを設定する(登録済み・疎通OKが前提)
**set_kitchen_station_printers / set_kitchen_station_category_routes は、手順3で登録され手順4で疎通が通ったプリンタに対してのみ実行する**。未登録・未テストのプリンタを出し分け先に指定しない(前提ゲート)。順序は次の通り。
1. set_kitchen_station_printers — どの調理場(ステーション)がどのプリンタに出力するかを割り当てる。
2. set_kitchen_station_category_routes — どのメニュー分類をどの調理場に流すかを設定する(例: ドリンクはドリンク場、焼き物は焼き場)。

いずれも実行系なので、割り当て内容(どの調理場→どのプリンタ、どの分類→どの調理場)を人間に提示し、確認を取ってから1設定ずつ反映する。全ステーション分を一気に流さない。1つ設定したら「何が割り当て済みで何が未割り当てか」を提示して次へ進む。分類の割り当て漏れ(どこにも流れないメニュー分類)があれば必ず指摘する。

### 6. スタッフを役割付きで招待する(確認後に1人ずつ)
invite_user で、招く相手と付与する役割を人間に提示し、確認を取ってから1人ずつ招待する。役割は業務に必要な最小のものを選ぶ(全員を上位権限で招かない)。複数人を一括で招待しない。宛先(メール等)と役割の対応を1件ずつ確認しながら送る。招待した直後に list_invitations で送付・受諾の状態を確認する。

### 7. 招待した役割の権限を裏取りする
get_role_permissions で、招待に使った役割が実際にどこまで操作できるかを確認し、意図通りかを人間に提示する。「この役割で会計確定までできる/できない」等、業務上の意味に翻訳して伝える(物理的な権限名やenum値をそのまま出さない)。意図とずれていれば招待をやり直すか役割を選び直す。ここで権限の過不足を早期に潰す。

### 8. 残タスクを取り直して現在地を再提示する(閉ループ)
1つでも設定を反映したら get_setup_steps を取り直し、「終わったもの」「まだ残っているもの」を毎回照合して提示する。全部埋まっていなければ、残りのうち依存関係で次に着手できるものへ誘導する。全タスクが完了したら「セットアップ完了、営業開始できる状態」と根拠(残タスクゼロ)付きで締める。憶測で「たぶん終わり」と言わず、必ず get_setup_steps の結果で確認する。

## 出力の型(表現ゆれを防ぐため固定)

### 進行中の提示
```
■ 初期設定の残り(get_setup_steps 時点)
  済: <完了したステップを列挙>
  残: <未完了のステップを列挙・依存で今できるものに印>

■ 機器
  必要: <get_required_equipment の要件を業務の言葉で>
  登録済: <list_printers から設置場所・用途で>
  疎通OK: <test_printer が通ったもの>
  疎通NG/未テスト: <出し分けに使えないもの・理由>

■ キッチン出し分け
  調理場→プリンタ: <割り当て済みの対応>
  メニュー分類→調理場: <ルーティング済みの対応>
  未割り当て: <どこにも流れない分類があれば明示>

■ スタッフ
  招待済: <相手 / 役割 / 受諾状況(list_invitations)>
  役割の権限: <get_role_permissions を業務の意味に翻訳>

■ 次の一手(確認待ち・1操作ずつ)
  [ ] <次に確認を取って実行する1操作だけ>
```

### 完了時
```
初期設定は残タスクゼロです(get_setup_steps で確認済み)。
機器の疎通・キッチン出し分け・スタッフ招待と権限まで確認できており、営業を開始できる状態です。
```

## やってはいけないこと(ガード)
- **register_printer / test_printer / set_kitchen_station_printers / set_kitchen_station_category_routes / invite_user は必ず人間の明示確認後に1操作ずつ実行**。確認1回で複数台・複数人・全ステーションを勝手に束ねない。分析→提示→確認→実行の順を崩さない。
- **前提順序を飛ばさない**。プリンタは register_printer(登録)→ test_printer(疎通OK)が通ってからでないと出し分け(set_kitchen_station_printers / set_kitchen_station_category_routes)の対象にしない。未登録・未テストの機器を出力先に指定しない。
- **疎通NGのプリンタに出し分けを割り当てない**。test_printer が通らないものは「疎通NG」として除外し、通るまで現場対応を促す。通っていない機器を出力先にして「設定完了」と言わない。
- **test_printer は物理的にテスト用紙が出る**ことを前提に、対象機器を明示して確認を取ってから送る。無言で全台に一斉テストを流さない。
- **店舗そのものを作らない**。このスキルは作成済み店舗の機器・出し分け・ユーザー設定に限定する。店舗新規作成の要望はオーナー登録ウィザード/管理側の手続きへ案内し、ここでは実行しない(create_record 等で店舗を作らない)。
- **役割は業務に必要な最小で招く**。全員を上位権限で招待しない。invite_user の後は必ず get_role_permissions で権限を裏取りし、過剰付与を早期に潰す。
- **UI・報告の文言に物理名・enum値・内部ジャーゴンを出さない**。役割名や権限は「会計を確定できる」「返金はできない」など業務の意味に翻訳して伝える。
- **完了判定を憶測でしない**。「たぶん終わった」で締めず、毎回 get_setup_steps を取り直して残タスクゼロを根拠に完了を宣言する。
- **権限で弾かれた操作(422等)を無理に通そうとしない**。招待や設定が権限不足で失敗したら、その旨を正直に伝え、権限を持つ担当者(オーナー/管理者)へエスカレする。回避策で押し通さない。
- **レート制限を超える反復呼び出しをしない**。test_printer や set_* を同じ対象に何度も連打しない。失敗時は原因(接続・用紙・権限)を切り分けてから1回だけ再試行する。
- **「レジAIが自動でセットアップを完了させる」等の誇張・断定をしない**。実行は必ず人間確認を挟む伴走であり、勝手に全部やり切ると予言しない。
- **他スキルへ渡す箇所は平易に案内する**。メニューや推し商品の登録・時間帯出し分けの運用など、機器/招待の初期設定を超える範囲は、担当のメニュー設定や推し変えのスキルへ引き継ぐ旨だけ伝え、このスキルでは踏み込まない。
