# 季節・高粗利プロモ差し替え(旗艦)

## これは何か
店長が「今週なに推す?」と聞いたとき、勘で1品挙げて終わりにしない。今の売れ方(スター/金のなる木)と時間帯の山、在庫と併売の実データから候補を絞り、押した結果の期待粗利を数字で見せてからオーナーに承認を仰ぐ。差し替えて終わりではなく、時間帯で出し分け、数日後に本当に効いたかを測って、効かなければ次候補へロールする。推し変えを「勘の芸」でなく「閉ループの運用」に変えるための旗艦スキル。

## 使うMCPツール(rocky-regi)
- get_menu_engineering — 4象限分類(スター/金のなる木/花形/負け犬)。押す候補の入口。主軸。
- get_menu_abc_analysis — 売上寄与のABC分類。粗利率実数で並べ替えるための素材。
- list_dead_menu_items — 死に筋の除外リスト(候補に混ぜない)。
- get_hourly_heatmap — 曜日×時間帯の山。昼枠/夜枠のマッピングに使う。
- list_inventory — 欠品・時価商品の足切り。押せない品を弾く。
- simulate_price_change — 読取専用。押した結果の期待粗利増を試算(需要予測ではない)。
- get_basket_analysis — 併売リフト。推し品と一緒に出すべき隣を決める。
- get_active_visibility_now — 今この瞬間の出方確認。予約が効いてるか。
- get_campaign_roi — lift/holdoutで効果測定。閉ループの締め。
- create_store_promo / update_store_promo — 推し登録・更新(実行系・要承認)。
- reorder_menu_items — カルーセル並び替え(実行系・要承認)。
- create_visibility_schedule / update_visibility_schedule — 時間帯出し分け予約(実行系・要承認)。

## 手順(この順で多段に呼ぶ)
### 1. 押す候補を4象限で抽出する
get_menu_engineeringで「スター(高人気×高粗利)=更に押す」「金のなる木(高粗利×低人気)=露出不足の可能性、押せば化ける」を拾う。花形(高人気×低粗利)は候補から外す(押しても粗利が上がらない)。負け犬は無視。

### 2. 死に筋を除外して粗利率で並べる
get_menu_abc_analysisで売上寄与Cランクの中身を確認、list_dead_menu_itemsに載っているものを候補から落とす。残った候補を粗利率の実数で並べ替える(勘の順位でなく数字の順位)。

### 3. 時間帯にマッピングする
get_hourly_heatmapで曜日×時間帯の山を見て、昼のピークには軽め・低単価寄り、夜のピークには高単価・高粗利を割り当てる。ピークが2つある店(ランチ/ディナー)は必ず別候補を用意する。ピークが1つしかない店は無理に分けない。

### 4. 在庫で足切りする
list_inventoryで欠品・在庫僅少・時価商品を確認。押した瞬間に「売り切れ」「時価で答えられない」となる品は候補から落とす。ここで候補ゼロになったら「今週は推し変えせず現状維持」と正直に返す。無理に押さない。

### 5. 期待粗利を試算して見せる(勘に見せない)
残った上位候補についてsimulate_price_changeを読取専用で回し、「この推し方でどれくらい粗利が乗るか」の試算を明示提示する。simulate_price_changeは需要予測ではなく値上げ耐性の確認である旨を必ず添える(「上げても離れにくいから押す価値がある」の裏取り、未来を予言するものではない)。

### 6. オーナー承認を取ってから差し替え実行
上位候補と期待粗利試算をオーナーに提示し、明示的な承認を取る。承認が出た1操作だけcreate_store_promo(または update_store_promo)で登録する。バッジ・画像・文面は業態に合わせる(居酒屋なら「今週の一杯」、カフェなら「季節限定」等、既存トーンに寄せる)。承認なしで先走らない。

### 7. アップセル分岐(併売品を隣に並べる)
get_basket_analysisで推し品と高リフトで併売される品を拾い、reorder_menu_itemsでカルーセル上の並びを「推し→高リフト併売」の順にする。ここも実行系なので承認後に1操作。並び替え不要なら省略する(無理に動かさない)。

### 8. 時間帯出し分けを予約する
create_visibility_scheduleで昼枠(軽め候補)/夜枠(高単価候補)を時間帯予約する。ピークが1つしかない店では単一枠でよい。予約系も実行系なので承認後に1操作ずつ。

### 9. 今この瞬間の出方を検証する
予約直後にget_active_visibility_nowで「今何が出ているか」を確認する。予約が意図通りに反映されているか、時間帯を跨いだ切替が起きているかを目視で裏取りする。反映されていなければ設定を見直す(次の予約は打たない)。

### 10. 数日後に効果測定して次候補へロール(閉ループ)
数日〜1週間後にget_campaign_roiでlift/holdoutを見る。効いていれば継続、効かなければ手順1で控えていた次候補へロールする。効果ゼロなのに惰性で残さない。ここで初めて閉ループが閉じる。

## 出力の型(表現ゆれを防ぐため固定)
1. **今週の推し候補(粗利順・上位3)**
   - 候補1: 商品名 / 象限(スター or 金のなる木) / 粗利率 / 期待粗利増(simulate_price_change試算)
   - 候補2: 同上
   - 候補3: 同上
2. **時間帯マッピング**
   - 昼枠(◯◯時〜◯◯時): 候補◯
   - 夜枠(◯◯時〜◯◯時): 候補◯
   - (ピーク1つの店の場合は単一枠と明記)
3. **足切りした品(理由付き)**
   - ◯◯: 死に筋のため除外
   - ◯◯: 在庫僅少のため除外
   - ◯◯: 時価商品のため除外
4. **併売の隣に置く品**
   - 推し◯◯ と一緒に出す: △△(リフト◯◯倍)
5. **実行プラン(承認待ち・1操作ずつ)**
   - [ ] create_store_promo: ◯◯を推しに登録
   - [ ] reorder_menu_items: カルーセルを◯◯→△△の順に

6. **効果検証の予定**
   - ◯月◯日にget_campaign_roiで測定 → 効かなければ次候補◯◯へロール

リッチUIで見せるときはshow_menu_engineering(4象限)とshow_hourly_heatmap(時間帯の山)を添える。数字だけで長文にしない。

## やってはいけないこと(ガード)
- **create_store_promo / update_store_promo / reorder_menu_items / create_visibility_schedule / update_visibility_schedule は必ずオーナー承認後に1操作だけ実行**。承認1回で複数操作を勝手に束ねない。固定文言でのゲートを守る。
- **simulate_price_changeを「需要予測」と呼ばない**。あくまで値上げ耐性の確認、押した結果の粗利試算に留める。未来の売上を予言しない。
- **候補ゼロを無理に埋めない**。在庫足切り後に候補が残らなければ「今週は推し変えせず現状維持」と正直に返す。作り話の候補を出さない。
- **死に筋を「隠れた名品」と言い換えない**。list_dead_menu_itemsに載っている品は候補から落とす。粗利率が高くても在庫が動かないなら押す土俵に上げない。
- **効いていない推しを惰性で残さない**。get_campaign_roiのliftがholdout比でプラスに出なければ次候補へロールする。「もう少し様子見」で握らない。
- **画像素材は人間publish承認**。create_line_message_asset_upload_url等で外部AIが生成した画像をそのまま本番反映しない。オーナー目視の後に投入。
- **レート制限を超えて反復呼び出ししない**。同じsimulate_price_changeを候補ごとに走らせるのは可、閾値超えの探索的スイープはしない。
- **ポータル巡回(食べログ/Google等の外部露出調整)はこのスキルで扱わない**。#1674系の外部ポータル自動巡回はBOT規約リスクで現在保留、店内(rocky-regi内)の推し差し替えに限定する。
- **Meta Ads等の外部広告連携は未接続**。契約後に.mcp.jsonへ追加される想定で、現時点では外部広告出稿は本スキルの範囲外。店内メニュー露出の差し替えのみ行う。
- **実行系(create_/update_/reorder_)と分析系を混ぜて一気に走らせない**。分析→提示→承認→実行の順を崩さない。SSRで気の利いた文面を先に流さない。
