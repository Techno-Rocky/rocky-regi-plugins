# 印刷が出ない時の切り分け

## これは何か
「伝票が出ない」「キッチン票が届かない」と言われたとき、とりあえず再送ボタンを連打しない。まず出力ジョブの状態と、失敗したジョブに付いているエラーの型を見る。エラーの型によって原因はまったく別物で、打つべき手も逆になる。プリンタに届いているのに印字できないのか(用紙切れ・紙詰まり)、そもそも届いていないのか(ネットワーク断)、一過性の瞬断で放っておけば直るのか、を型で切り分けてから、用紙・ネットワーク・設定のどれか1つに原因を寄せる。物理原因なら現場対応が先、再送はその後。設定変更は運用に影響するので店長・管理者の確認を挟む。連打で紙とインクを溶かさず、真因に最短で寄せるための診断スキル。

## 使うMCPツール(rocky-regi)
- list_print_jobs — 読取専用。出力ジョブの状態とエラー文字列/HTTP相当コードを取る。診断の主軸。ここでエラーの型が決まる。
- show_print_jobs — 読取専用。ジョブ一覧をリッチUIで見せる。切り分けの根拠を人間に示すとき。
- list_printers — 読取専用。対象プリンタの登録・オンライン状態・割り当てを確認。どのプリンタの話かを固定する。
- get_print_transport — 読取専用。現在の転送方式(印刷の届け方)を確認。届いていない型のとき方式を疑う材料。
- get_required_equipment — 読取専用。この店で必要な機器と、初期設定が済んでいるかの確認。未設定起因の切り分けに使う。
- test_printer — 実行系(要確認)。対象プリンタへ疎通テストを1台ずつ送る。届くかどうかの生の確認。
- retry_print_job — 実行系(要確認)。失敗ジョブを再送する。原因を潰した「後」に1件ずつ。
- set_print_transport — 実行系(要確認・実質不可逆に近い設定変更)。転送方式を切り替える。店長/管理者の確認が要る。現場スタッフ権限では弾かれ得る。

## 手順(この順で多段に呼ぶ)

### 1. 対象を1つに固定する
「どの伝票が」「どのプリンタで」出ないのかを1つに絞る。レシート(会計伝票)なのか、キッチン票なのか、注文確認票なのかで見るジョブが違う。プリンタが複数ある店では list_printers で対象プリンタを特定し、オンライン状態と割り当てを確認する。ここで「そもそもプリンタが未登録/オフライン」なら、それが答えで、再送以前の話。複数プリンタをまとめて扱わず、症状の出ている1台に寄せる。

### 2. まずジョブの状態とエラーの型を読む(ここが核心)
何をおいても先に list_print_jobs を読む。test_printer や retry_print_job に走る前に、必ずここでエラー文字列とHTTP相当コードを確認する。この型を見ずに再送するのが一番やってはいけないこと。根拠を人間に示すときは show_print_jobs でリッチに開く。ジョブが「成功」で並んでいるのに現物が無い場合は、プリンタの物理側(用紙・電源・カバー)をまず疑う。

### 3. エラーの型で原因を切り分ける(診断木・分岐)
list_print_jobs のエラーを次の型に当てはめる。型ごとに寄せる先が違う。

- **"EX_TIMEOUT" かつ HTTP 200 相当** = プリンタには届いているが印字できていない。**用紙切れ/紙詰まりが濃厚**。ここでネットワークは疑わない。手順4(物理確認)へ。届いているので接続や転送方式をいじるのは筋違い。
- **"Failed to fetch" / HTTP 0 相当** = そもそも届いていない。**ネットワーク断**。ルータ・同一LAN・プリンタの電源を確認。Android/Chrome端末では、初期設定でローカルネットワークアクセス権限が未許可だとこの型になる(端末アドレスバーの鍵アイコン > ローカルデバイスの検索/アクセスを許可)。手順5(ネットワーク確認)へ。**接続が戻る前に retry を連打しても無駄**、復旧が先。
- **"claim jobs failed" 系の一過性 fetch 失敗** = 良性の瞬断であることが多い。次のポーリングで自然に回復する。**すぐ再送・再設定に走らず一拍おく**。1回きりなら様子見。多発・継続する時だけ手順5で接続を疑う。単発をネットワーク障害と決めつけない。
- **上記に当てはまらない/エラー無しで成功済みなのに出ない** = 物理側(用紙・電源・カバー開)か、そもそも印刷指示が飛んでいない可能性。手順4で物理を確認しつつ、対象プリンタ・割り当てが正しいかを list_printers で見直す。

### 4. 物理原因の型なら、現場対応を先に案内する(EX_TIMEOUT系)
用紙切れ・紙詰まり・カバー開・電源は、ツールでは直せない。**先に現場スタッフへ「用紙残量・紙詰まり・ローラーカバー・電源」の確認を案内**する。ここを飛ばして retry_print_job を送っても、同じ EX_TIMEOUT でまた失敗し、待たせるだけ。用紙を補充した/詰まりを除去したと現場から確認が取れて初めて、手順7の再送に進む。

### 5. 届いていない型なら、接続復旧を先に(Failed to fetch系)
ネットワーク断は復旧が先で再送は後。同一LAN・ルータ・プリンタ電源・端末のローカルネットワークアクセス権限を現場に確認してもらう。切り分けの生確認が要るときだけ test_printer を使う(手順6)。get_print_transport で現在の転送方式を読み、届け方が想定と違う疑いがあるときは手順8で設定変更を検討するが、これは店長/管理者マター。初期設定そのものが未了の疑いなら get_required_equipment で必要機器と設定状況を確認する。

### 6. 疎通確認が要るときだけ test_printer(実行系・要確認・1台ずつ)
「復旧したはず」「本当に届くのか」を生で確かめたいときに test_printer を使う。実行系なので、対象プリンタ名を人間に提示して確認を取ってから、1台ずつ送る。複数台を一括でテストしない。テストが通れば接続は生きている(残る原因は物理か指示側)、通らなければまだネットワーク側が生きていない。テスト結果を根拠に、手順7の再送に進むか、手順5の接続確認に戻る。

### 7. 原因を潰した後に retry_print_job(実行系・要確認・1件ずつ)
再送は「原因を潰した後」に打つのが鉄則。用紙を補充した/詰まりを取った/接続が戻ったことを確認してから、失敗ジョブを1件ずつ retry_print_job で再送する。対象ジョブと、何が原因でどう潰したかを人間に提示し、確認を取ってから送る。**失敗ジョブを全件一括で再送しない**(原因未解消のまま束で再送すると同じ失敗を量産する)。1件通ったのを確認してから次へ。原因が用紙・接続のどちらでもなく型が読めないままなら、再送でなく手順8か、設定担当・管理者へのエスカレに倒す。

### 8. 転送方式の変更は店長/管理者の確認ゲート(set_print_transport・不可逆に近い)
届け方(転送方式)そのものを疑う段になったら、get_print_transport で現在値を読むところまではしてよい(読取専用)。切り替え=set_print_transport は運用全体に影響する設定変更で、実質不可逆に近い。**必ず、対象プリンタ・現在の方式・変更後の方式・影響範囲・元に戻せるかを提示し、店長または管理者の明示的な確認を取ってから、1操作だけ実行**する。現場スタッフ権限では権限エラー(422相当)で弾かれ得る。その場合は無理に通そうとせず、設定担当・管理者へエスカレする。切り分けが終わっていないのに転送方式から先にいじらない。

## 出力の型(表現ゆれを防ぐため固定)
1. **症状の特定**
   - 出ないもの: レシート / キッチン票 / 注文確認票 のいずれか
   - 対象プリンタ: ◯◯(オンライン状態・割り当て)
2. **ジョブ状態とエラーの型**
   - 直近失敗ジョブ: ◯件
   - エラー文字列 / HTTP相当: 例)"EX_TIMEOUT" / 200
   - 判定した型: 用紙(EX_TIMEOUT+200) / ネットワーク断(Failed to fetch+0) / 一過性瞬断(claim jobs failed) / 型不明
3. **切り分け結論(原因を1つに寄せる)**
   - 原因: 用紙切れ・紙詰まり / ネットワーク断 / 良性の瞬断(様子見) / 設定(転送方式) / 未確定
   - 根拠: 型と数字を1〜2行で
4. **先にやること(現場 or 復旧)**
   - 物理型: 用紙・詰まり・カバー・電源を現場確認
   - ネットワーク型: 同一LAN・ルータ・電源・ローカルネットワーク権限を確認
   - 瞬断型: 一拍おいて次のポーリングを待つ(再送しない)
5. **実行プラン(確認待ち・1操作ずつ)**
   - [ ] test_printer: ◯◯へ疎通テスト(確認後・1台)
   - [ ] retry_print_job: 失敗ジョブ◯を再送(原因を潰した後・1件)
   - [ ] set_print_transport: 方式を◯◯へ変更(店長/管理者確認後・1操作)※不可逆に近い

診断の根拠を見せるときは show_print_jobs でジョブ一覧を開く。エラーの型が読めているうちは、テキストで型と寄せ先を先に示す。

## やってはいけないこと(ガード)
- **list_print_jobs でエラーの型を読む前に retry_print_job / test_printer を叩かない**。型を見ずの再送が最悪手。まず状態→型→寄せ先の順。
- **EX_TIMEOUT+200(届いている型)でネットワークや転送方式を疑わない**。これは用紙・詰まりの型。物理確認が先で、接続や set_print_transport をいじるのは筋違い。
- **Failed to fetch/HTTP0(届いていない型)で先に retry を連打しない**。接続復旧が先、再送は後。Androidはローカルネットワークアクセス権限の未許可を必ず候補に入れる。
- **claim jobs failed の単発を障害と決めつけない**。良性の瞬断のことが多く、一拍おけば回復する。すぐ再設定・再送に走らない。多発・継続のときだけ接続を疑う。
- **失敗ジョブを全件一括で再送しない**。原因を潰した後に1件ずつ。束の再送は同じ失敗を量産する。
- **test_printer / retry_print_job / set_print_transport は人間の明示確認後に1操作ずつ**。分析→提示→確認→実行の順を崩さない。読取(list/get/show)と実行を混ぜて一気に走らせない。
- **set_print_transport は不可逆に近い設定変更として扱う**。対象・現在値・変更後・影響・戻せるかを提示し、店長/管理者の確認を取ってから1操作だけ。切り分け前に転送方式から先にいじらない。
- **権限で弾かれる操作(422相当)を無理に通さない**。現場スタッフ権限で set_print_transport 等が弾かれたら、設定担当・管理者へエスカレする。回避策で押し通さない。
- **物理原因をツールで解決しようとしない**。用紙切れ・紙詰まりは補充・詰まり除去という現場対応が先。ツールの再送はその後。
- **静的な画面や成功表示だけで「直った」と判断しない**。EX_TIMEOUT は HTTP は成功相当に見えることがある。現物が出たか、現場の確認を根拠にする。
- **レート制限を超えて test_printer / retry_print_job を反復しない**。疎通テストも再送も、確認しながら小刻みに。連打で溢れさせない。
- **「レジAIが自動で復旧する」等の誇張・断定をしない**。このスキルは切り分けと案内まで。物理対応と設定変更は人間の手と確認に返す。
- 転送方式そのものの詳細設計や、機器の新規登録・初期セットアップは本スキルの範囲外。必要なら初期設定・機器登録の担当フローへ、平易な案内で渡す。
