# 離反客の掘り起こしLINEシナリオ設計(方向で組む)

## これは何か
「最近来てない客にLINEを打ちたい」と言われたときに、思いつきで一斉配信して終わらせない。誰が本当に「離反しかけた優良客」なのかをデータで定義し、そもそも今月そのセグメントに何通打てる余力があるのかを配信前に確かめ、戻ってきたときに粗利が取れるオファーを設計し、holdoutを最初から切ってROIが後日測れる形に落とす。ロッキーレジ本部CRMの掘り起こし施策の起点であり、着地点。

## 使うMCPツール(rocky-regi)
- get_customer_segments_summary — 店の顧客をRFMで俯瞰。離反リスクセグメントの規模感を最初に押さえる主軸。
- get_customer_metrics — 顧客ごとのR/F/M実数値。閾値を勘でなく分布から決めるための素材。
- get_customer_context / list_customer_visits — 個別顧客の来店間隔・直近来店の実分布。閾値の妥当性を裏取り。
- preview_line_segment — 定義したセグメントがLINE友達として実際に何人到達可能かを配信前に見る。このフロー固有の価値。
- get_line_messaging_quota — 今月のLINE配信残枠。打てる規模かの上限判定。
- get_basket_analysis / get_menu_engineering — 戻ったときに併売で粗利が取れる高粗利商品の特定。オファー設計の材料。
- create_coupon_distribution_scenario — holdout群を最初から切る指定でシナリオ化(書込・オーナー承認必須)。
- set_coupon_distribution_scenario_enabled — 承認後の配信開始スイッチ(書込)。
- get_campaign_roi / get_intervention_effect — 配信後のROI測定(lift/holdout比較)。当フローでは「後日の再訪ステップ」として言及のみ。
- list_stores — 対象店の確定(複数店運用時・手順1)。

## 手順(この順で多段に呼ぶ)

### 1. 対象店と分析期間を決める
list_storesで対象店を確定させる。分析期間は直近6ヶ月をデフォルト、業態(居酒屋/カフェ/クリニック等)で来店周期が違うので店側から違和感が出たら伸縮する。ここで期間を勝手に長く取ると休眠層と離反しかけ層が混ざる、短く取るとFの実態が見えない、と伝える。

### 2. RFMで「離反しかけた優良客」を定義する
get_customer_segments_summaryで全体の分布を掴んだうえで、get_customer_metricsで顧客個別のR/F/Mを取る。ここで狙うのは「休眠客(F低・M低)」でも「今のロイヤル(R良好)」でもなく、「R悪化 × F/M高い」=戻ってくれば粗利で効く層。この層に絞る理由を店側に一言添える。

### 3. 閾値をデータ由来で決める(勘で切らない)
上位優良客(F・M上位)の平均来店間隔をget_customer_context/list_customer_visitsから出し、その1.5倍を超えているRを「離反しかけ」の境界とする。「先月来てないから離反」等の勘定義は使わない。閾値の根拠(上位客平均◯日 × 1.5 = 境界◯日)を必ず出力に載せる。

### 4. 到達検証を配信前に走らせる(このフローの心臓部)
定義したセグメントをpreview_line_segmentに渡し、LINE友達として実際に届く人数を出す。同時にget_line_messaging_quotaで今月の残quotaを取る。ここで「セグメント該当◯人 / LINE到達◯人 / 今月残枠◯通」を並べ、打てる規模かを判定する。到達人数が2桁前半しかない/残枠が足りない場合はここで止めて店側に伝え、閾値を緩めるかセグメントを広げるか判断してもらう。この検証を飛ばして施策設計に進まない。

### 5. 戻ったとき粗利が取れるオファーを設計する
get_basket_analysisで対象セグメントが過去に併売してきた組み合わせを、get_menu_engineeringで店の高粗利メニュー(スター/金のなる木)を確認し、両者が重なるものをクーポン対象に置く。原価率が高い集客商品を割引対象に選ばない。割引率は「戻り客一人あたり期待粗利 > クーポン原価」で成立する範囲に収める。ここは店側と一度合意する。

### 6. holdout付きシナリオを組み立て、オーナー承認を取る
create_coupon_distribution_scenarioでシナリオを作成する際、holdout群を最初から切る指定を必須とする(切らないならROIは名目倒れ、と店側に伝える)。holdout比率は到達人数から逆算(到達が少ないなら10%、余裕があれば20%目安)。作成はドラフト状態にとどめ、次の「出力の型」を提示してオーナーの明示承認を得たあとにset_coupon_distribution_scenario_enabledで配信を開始する。オーナー承認前に有効化しない。

### 7. 後日のROI測定計画を添える(実行はしない)
配信終了後の測定として、get_campaign_roi(lift/holdout)とget_intervention_effectで戻り客数・粗利lift・holdout比較を見る、と一文だけ添える。当フローでは実行せず、後日の別セッションに渡す。

## 出力の型(表現ゆれを防ぐため固定)
show_customer_cardを使えるならリッチUIで顧客サンプルを見せ、本体は以下の固定カードで承認判断できるようにする。

- 対象セグメント定義:上位優良客の平均来店間隔◯日 × 1.5 = ◯日超未来店 / F上位◯割 / M上位◯割
- 該当人数:◯人(全顧客に対する割合:◯%)
- LINE到達人数:◯人(セグメントのうちLINE友達比率:◯%)
- 今月のLINE残quota:◯通(この施策で使う想定:◯通 / 残り:◯通)
- オファー内容:対象メニュー(高粗利かつ併売実績あり)/割引率◯% / 有効期限◯日
- 期待戻り客数:◯人(過去の同種lift実績があれば参考値として、なければ「実績なし・holdoutで測定」と明記)
- 期待粗利:◯円(戻り客一人あたり期待粗利 × 期待戻り客数 − クーポン原価)
- holdout設計:比率◯%(◯人)/測定はget_campaign_roiで後日実施
- 配信予定日時:◯月◯日 ◯時
- 承認要否:オーナー明示承認後にset_coupon_distribution_scenario_enabledで有効化

このカードのうちどれか一つでも欠けたまま有効化に進まない。

## やってはいけないこと(ガード)
- 閾値を勘で切らない。「最近来てない=1ヶ月来てない」等の直感定義は使わない。必ずget_customer_metrics/list_customer_visitsの実分布から出す。
- preview_line_segmentとget_line_messaging_quotaを飛ばして施策設計に入らない。届かない相手に設計しても意味がなく、残枠不足で当月配信できない事故を防ぐため。
- holdoutを切らないシナリオを作らない。切らないならROIは名目倒れで、後でget_campaign_roiを回しても答えが出ない。到達が少なくて統計的に厳しい場合はその旨を出力に明記する(「holdout◯人・効果測定は参考値」)。
- オーナーの明示承認前にset_coupon_distribution_scenario_enabledを叩かない。create_coupon_distribution_scenarioまでで止め、固定カードを提示して承認を待つ。書込系・実行系は必ず人間のpublish承認を挟む。
- 原価率の高い集客商品を割引対象に置かない。get_menu_engineeringのスター/金のなる木と、get_basket_analysisの併売実績が重なる商品に絞る。
- 期待戻り客数・期待粗利を断定調で書かない。過去lift実績がなければ「実績なし・holdoutで測定」と正直に添える。SSR的な予言を出さない。
- レート制限を超える反復呼び出しをしない。get_customer_metricsを全顧客ぶんループで回す前に、まずget_customer_segments_summaryで規模を絞る。
- 「レジがAIで自動的に離反客を呼び戻す」といった誇張表現を店側への説明に使わない。あくまで「外部AIから叩ける土台で、離反セグメント抽出→到達検証→holdout付き配信→ROI測定を一本で回す」までしか言わない。
- 効果測定(get_campaign_roi/get_intervention_effect)を当フロー内で回さない。配信直後にlift評価を出しても意味がなく、別セッション(後日)に渡す。
- Meta広告など外部配信チャネルは当フローの対象外。Meta公式Ads MCPは未接続の想定なので、LINE以外への横展開を勝手に前提化しない。
